> 最后更新: 2026-06-29
> 当前代码事实优先于早期 v0.1.0 基线文档。

## 截图追溯 P0 基线

2026-06-28，用户再次贴出前序复审截图并指出：“这里面的内容 你是不是已经完全丢失 忘记了，我觉得这些也是 P0 级别的内容”。因此下面内容是当前 P0 规划和验收来源，不得在后续推进中降级为 P1/P2，也不得因为先推进 Artifact、ActionRun 或单个页面切片而被弱化。

- **开箱体验与工作系统金线**：Gateway 不能被隐藏，但必须从“用户目标”降级为“基础设施”；第一感受应是“创建我的工作系统”，不是“连接控制台”。
- **自由边界**：保持自由、不被第三方商业生态绑定是 P0 产品约束；外部生态入口可以暂不作为当前代码实施阻塞，但不能把“自由”从 P0 验收中移走。
- **Dashboard 真实推进状态**：首屏应优先回答今日继续、待确认、卡住事项、最近成果、知识动态和本周新增成果；Gateway 健康状态只是基础设施状态。
- **截图细化验收**：开箱金线必须显式覆盖“打开应用 -> 选择语言/主题 -> 自动发现或安装 Gateway -> 创建本地工作仓库 -> 输入第一件事 -> 进入工作台”；事务推进必须显式覆盖“事项唯一 ID -> ActionRun 归属事项 -> 写入运行记录 -> 回写事项页 -> 触发更新状态/沉淀成果/更新知识库/写入复盘尾动作”。
- **Repository 初始化与普通人语言包**：首次连接成功后应引导创建本地工作仓库；UI 第一层优先使用资料、知识、事项、计划、执行记录、成果、复盘。
- **Knowledge 导入/消化/健康检查**：导入中心、未消化资料队列、AI 消化计划、写入 `wiki/` / `wiki/index.md` / `wiki/log.md`、健康检查和 `reviews/weekly/` 归档都属于 P0 闭环。
- **事务推进闭环**：工作事项要有唯一 ID，ActionRun 要归属事项，执行记录要写回事项页，结束后要触发更新状态、沉淀成果、更新知识库、写入复盘等尾动作。
- **开始一件事闭环**：用户一句话 -> 事项 -> 计划 -> 执行 -> 产物 -> 复盘是专题级 P0，不是体验优化。
- **可复用资产一等对象**：工具、脚本、模板、工作流、提示词、检查清单等需要来源、版本、权限、审批边界、运行记录和产出关联。

说明：外部生态入口（SkillHub / ClawHub / GitHub release 等）当前不作为本阶段具体代码实施阻塞；这不等于自由边界不是 P0。P0 验收仍必须持续确认用户可通过开源源码、协议开放、数据可迁移和仓库事实源保持自由，不被任何第三方商业生态绑定。

## 当前推进补充

- **Artifacts 普通魔法创建发起前事项选择**：Artifacts 页面普通“AI 魔法创建”入口在没有外部来源事项时，已读取当前绑定 Repository 的 Workbench Snapshot，从 `work/active`、`work/someday` 和 `work/completed` 列出已有事项；用户可在发起 `artifact_create` ActionRun 前选择关联事项，Desktop 会把事项 `workItemPath` 和 frontmatter `id` 写入运行记录。用户跳过选择时，运行仍按 `workItemRequired: true` 与 `workItemUnassignedReason: pending_work_item_assignment` 进入未归属诊断，不自动猜测归属。
- **Artifacts 普通魔法创建发起前即时创建事项**：同一入口在没有合适事项时，也可输入新事项标题并显式创建 `work/active/YYYY-MM-DD-HHmmss-*.md`；新事项使用 `source: desktop-action-run`，保留唯一 ID、目标、验收标准、执行记录、关联成果和复盘占位，创建后自动作为本次 `artifact_create` ActionRun 的 `workItemPath` / `workItemId`。该能力只服务用户显式创建，不自动猜测事项、不绕过 Repository Context、不替代后续计划/执行/产物/复盘金线；后续仍需把选择/创建模式提升为所有新 ActionRun 的全局体验。
- **Knowledge 普通自动改写发起前事项选择/创建**：事项候选加载和即时创建已抽为共享 `useWorkbenchWorkItemOptions` / `loadWorkbenchWorkItemOptions` / `createWorkbenchWorkItemOption` 能力；Knowledge 页面普通“消化资料 / 自动改写 / 刷新索引日志”入口在没有 Dashboard 尾动作来源事项时，会列出当前 Repository 的已有事项，也可输入新事项标题并显式创建 `work/active` 事项，随后在 `knowledge_rewrite` ActionRun 中写入 `workItemPath` 与 frontmatter `id`。Dashboard 知识尾动作仍以来源 URL 中的 `workItemPath` 为准。
- **Teams 自然语言编排发起前事项选择/创建**：Teams 页面自然语言编排和快速创建 Gateway Agent 两个非聊天式 AI 入口已复用共享事项选择/创建能力；用户可在发起 `agent_team_compose` 或 `gateway_agent_create` ActionRun 前选择已有事项，或显式创建新事项，Desktop 会写入 `workItemPath` 与 frontmatter `id`。用户跳过选择和创建时，运行仍进入未归属诊断；该入口不猜测归属、不替代 Repository Context 或审批。
- **Repository 语义映射发起前事项选择/创建**：RepositoryGate 的知识库语义映射和工作台语义映射入口也已复用共享事项选择/创建能力；用户可在发起 `knowledge_repository_map` 或 `workbench_repository_map` ActionRun 前选择已有事项，或在当前绑定仓库 ready 时显式创建新事项，Desktop 会写入 `workItemPath` 与 frontmatter `id`。该入口仍只做结构识别和映射保存，不猜测归属、不替代 Repository Context 或审批。
- **Knowledge/Teams/RepositoryGate 发起前即时创建事项**：这三个非聊天 ActionRun 入口现已与 Artifacts 一样具备“没有合适事项时新建并关联”的显式路径。剩余 P0 缺口不再是单入口能不能创建事项。
- **ActionRun 发起前事项面板组件化**：`ActionRunWorkItemPicker` 已把 Artifacts、Knowledge、Teams、RepositoryGate 四个入口的“选择已有事项 / 即时创建事项 / 创建后自动选中 / 成功失败提示”收敛为共享 UI 组件；事项候选和创建仍由 `useWorkbenchWorkItemOptions`、`loadWorkbenchWorkItemOptions`、`createWorkbenchWorkItemOption` 提供。当前完成的是四个现有入口的共享产品部件，不是未来所有新 ActionRun 的全局发起协议；后续仍需真正的全局 ActionRun 发起前面板，并继续推进计划、执行、产物、知识更新和复盘金线。
- **工作事项发起计划 ActionRun**：Workbench 预览工作事项时新增“生成计划”入口，创建 `work_matter_plan` ActionRun 并带入 `workItemPath` / `workItemId`；新增 `work-matter-plan.md` 和 `buildWorkMatterPlanPrompt`，要求 Agent 先读事项上下文再提出可审批计划。该入口只发起计划草案和审批方案，不静默写 `plans/active/`，不自动更新事项、沉淀成果、更新知识库或写复盘。
- **事项计划审批写入与互链**：`work_matter_plan` 的审批块现在要求结构化 `repositoryWrite`；Action Center 批准后会调用 `applyWorkbenchMatterPlanApproval`，仅允许写入当前绑定仓库 `plans/active/`，并把计划链接回来源事项 `## 关联计划`。该步骤只是审批后的计划落盘和互链，不代表计划已执行。
- **活跃计划发起执行 ActionRun**：Workbench 预览 `plans/active/` 计划时新增“执行计划”入口，创建 `plan_execute` ActionRun，并用 `plan-execute.md` / `buildPlanExecutePrompt` 带入计划内容和计划元数据中通过 `work/(active|completed|someday)/*.md` 边界校验的来源事项；终态后继续走既有 ActionRun 运行记录、事项执行记录和收尾动作机制。该入口不自动沉淀成果、更新知识库或写复盘。
- **计划执行状态观测**：Workbench 活跃计划列表和选中计划预览头部已通过 `findLatestPlanExecutionRun` 关联 `plan_execute` 输入中的 `planPath`，展示最近执行状态/摘要，并保留进入 Action Center 查看运行详情的入口；这只是观测，不自动改写计划、不自动沉淀成果、不更新知识库或写复盘。
- **计划执行成果沉淀入口**：Workbench 选中活跃计划时会通过 `shouldOfferPlanExecutionOutputPreservation` 判断最近一次 `plan_execute` 是否已完成、有 `resultSummary`、有安全 `workItemPath` 且没有 `artifactIds`；满足条件时显示“沉淀成果 / Preserve Output”，并用 `action-run-output:<runId>` 打开 Artifacts 的 `tailAction=output` 上下文。Artifacts 预填提示会保留来源事项和来源执行记录，用户显式保存后才写回来源事项 `## 关联成果`，保存后会触发 `notifyActionRunsChanged` 让 Dashboard 和 Workbench 重新加载本地 ActionRun 观察状态；Dashboard 现在也会把 `source.type=action_run` 且 `source.id=<runId>` 的 Artifact 视为该 run 已有产物承接，不再重复显示 `action-run:output-unpreserved`；该入口不自动创建产物、不更新知识库、不写复盘。
- **计划执行成果候选提取**：Artifacts 从 `action-run-output:<runId>` 进入成果沉淀时，会加载来源 ActionRun，并通过 `buildArtifactOutputPreservationPrompt` / `extractActionRunOutputCandidates` 把 `resultSummary`、`lastAssistantResponse`、`parseArtifactsFromText` 解析出的 `<artifact>` 块，以及显式输出段落中的候选成果带入 AI 创建初始提示，包括文件、链接、HTML、文档、表格和演示等线索。该能力只增强提示上下文，不自动创建 Artifact 或 Repository output、不读取任意本地文件、不执行文件、不授予权限。
- **AI 产物创建保存表单结构化 Artifact 块**：AI 创建保存表单已支持 `<artifact>`；`artifact-ai-create-preview` 会优先解析富结构 Artifact 块，再兼容旧 `ai-action` JSON，保存预览和 `generateArtifact` 参数会携带 HTML 正文、`externalFormat`、`contentSummary`、`reuseKind`、文件/链接元数据以及 `source.type=action_run`。该能力让 HTML、文档、表格、演示、链接等有价值成果不被压扁成工具/脚本线索，但仍只在用户显式保存后创建 Artifact，不自动写 Repository output、不执行资产、不更新知识库、不写复盘、不更新事项状态。
- **AI 产物创建保存前多候选选择**：AI 创建保存表单已支持多个候选产物；`parseArtifactAICreatePreviews` 会把一次响应里的多个 `<artifact>` 块保留为多个候选，`ArtifactAICreateDrawer` 会显示候选数量和候选按钮，用户选择其中一个后再显式保存。该能力不自动批量创建 Artifact、不自动写 Repository output、不执行资产、不更新知识库、不写复盘、不更新事项状态。
- **AI 产物创建保存前基础编辑**：AI 创建保存表单已支持保存前基础编辑；`ArtifactAICreateDrawer` 可在 selected candidate 上编辑标题、类型、说明、标签和价值摘要，`normalizeArtifactAICreatePreviewDraft` 会在保存前裁剪标题、说明和价值摘要首尾空白并移除空标签，空标题会阻止保存。该能力不编辑来源 ActionRun 或权限边界，不自动批量创建 Artifact、不自动写 Repository output、不执行资产、不更新知识库、不写复盘、不更新事项状态。
- **AI 产物创建保存前 HTML 正文编辑**：AI 创建保存表单已支持保存前编辑 HTML 正文；当 selected candidate 携带 HTML 正文或 HTML 格式线索时，`ArtifactAICreateDrawer` 会显示 HTML 正文编辑区，用户校正后的 HTML 会作为同一个 candidate 草稿传入 `buildArtifactAICreateGenerateParams`，`normalizeArtifactAICreatePreviewDraft` 会原样保留 HTML 正文，不裁剪正文首尾字符。该能力只改变用户显式保存前的 HTML 草稿，不自动检查或修复 HTML、不编辑文件路径、链接、来源 ActionRun 或权限边界，不自动批量创建 Artifact、不自动写 Repository output、不执行资产、不更新知识库、不写复盘、不更新事项状态。
- **AI 产物创建保存前文件/链接细节编辑**：AI 创建保存表单已支持保存前编辑文件/链接细节；`ArtifactAICreateDrawer` 可编辑 `externalFormat`、`reuseKind`、URL、命令、本地文件路径、文件名、文件大小、MIME 和“保存时导入本地文件副本”，`normalizeArtifactAICreatePreviewDraft` 会在保存前裁剪这些字符串元数据首尾空白，`buildArtifactAICreateGenerateParams` 会把校正后的细节传给 `generateArtifact`。该能力只改变用户显式保存前的 Artifact 元数据，不在编辑时读取本地文件、不打开链接、不执行命令、不授予权限、不编辑来源 ActionRun，不自动批量创建 Artifact、不自动写 Repository output、不更新知识库、不写复盘、不更新事项状态。
- **Artifact 可执行资产执行边界面板**：Artifact 详情页已能显示执行边界；当 Artifact 的 `reuseKind` 为 `tool / script / workflow` 时，页面会显示“执行前需审批”、最近审批/运行事实、`recordOnly` 硬边界，以及 `desktop.artifacts.execution.prepare` / `desktop.artifacts.execution.record` 命令线索。Desktop 只记录审批与执行事实，不执行资产、不授予权限；外部 runner 仍需用户显式审批。该能力只是把可复用资产的权限/审批边界产品化展示，不自动运行命令、不写复盘、不更新事项状态、不沉淀成果、不更新知识库。
- **Repository 资产执行记录与复盘入口**：仓库本地 `tool / script / workflow` 已可由 `desktop.repository.assets.execution.record` 归档外部 runner 的审批、运行或结果事实；Desktop 会写入 `runs/assets/YYYYMMDD-HHmmss-<assetId>.md`，维护 `runs/assets/index.md` 运行索引，并刷新 `outputs/assets/index.md` 的最近运行、lastRun 和复盘线索。传入当前仓库 `work/` 下安全 `workItemPath` 时，Desktop 还会回写来源事项 `## 执行记录`，并追加 `## 收尾动作` 的状态、成果、知识和复盘后续检查项。Workbench Snapshot 已把 `runs/assets/index.md` 合并进 Repository runs，Dashboard 会把资产运行结果作为最近成果/本周成果线索，并把 `review: pending` 显示为 `asset-run:review-pending` 待确认，目标 URL 携带 `assetRunPath` 和可选 `workItemPath`。Workbench reviews 已显示“资产运行复盘”卡片，可基于 `runs/assets/*.md` 创建 `reviews/weekly/YYYY-MM-DD-asset-run-*-review.md` 草稿；Gateway 也可调用 `desktop.repository.assets.execution.review.write` 写入同类草稿。用户显式确认草稿后，Desktop 会写入 `status: confirmed` / `reviewedAt`，Dashboard 会用 confirmed 复盘文档过滤同一资产运行的待确认提醒。该入口只记录、观察和写复盘记录，不执行资产、不授予权限、不更新事项状态、不自动沉淀成果、不更新知识库、不勾选事项尾动作。
- **计划执行知识更新入口**：Workbench 选中活跃计划时会通过 `shouldOfferPlanExecutionKnowledgeUpdate` 判断最近一次 `plan_execute` 是否已完成、有 `resultSummary` 且有安全 `workItemPath`；满足条件时显示“更新知识 / Update Knowledge”，并用 `action-run-knowledge:<runId>` 打开 Knowledge 的 `tailAction=knowledge` 上下文。Knowledge prompt 会保留来源事项和来源执行记录；因为 `action-run-knowledge:<runId>` 不是事项 checklist 尾动作 ID，该入口可发起 `knowledge_rewrite`，但不会显示或执行“确认已处理并完成尾动作”，也不会自动写 Wiki/index/log、更新状态、沉淀成果或写复盘。
- **计划执行复盘草稿入口**：Workbench 选中活跃计划时会通过 `shouldOfferPlanExecutionReview` 判断最近一次 `plan_execute` 是否已完成、有 `resultSummary` 且有安全 `workItemPath`；满足条件时显示“写复盘 / Write Review”，并用 `action-run-review:<runId>` 打开 Workbench reviews 的 `tailAction=review` 上下文。Workbench 可据此写入 `reviews/weekly/` 复盘草稿，在草稿中记录来源执行记录，并把草稿链接回来源事项 `## 复盘`；因为 `action-run-review:<runId>` 不是事项 checklist 尾动作 ID，该入口不会显示或执行“确认复盘并完成尾动作”，也不会自动勾选事项尾动作、更新状态、沉淀成果或更新知识库。
- **计划执行知识/复盘后续入口状态刷新**：Workbench 现在会把已有后续事实纳入 `shouldOfferPlanExecutionKnowledgeUpdate` / `shouldOfferPlanExecutionReview` 判断；如果同一来源事项下已有未失败/未取消的 `knowledge_rewrite` ActionRun，且输入包含 `action-run-knowledge:<runId>`，则不再重复显示“更新知识”；如果 `reviews/weekly/` 复盘文档已有同一 `workItemPath` 且 frontmatter 记录 `sourceExecutionId` 或 `tailActionId` 为 `action-run-review:<runId>`，则不再重复显示“写复盘”。该状态刷新只避免重复发起，不自动写 Wiki/index/log、不自动确认复盘、不勾选事项尾动作。
- **计划执行复盘草稿带入相关知识更新**：Workbench 创建 `action-run-review:<runId>` 来源的复盘草稿时，会查找同一来源事项下未失败/未取消且输入包含 `action-run-knowledge:<runId>` 的 `knowledge_rewrite` ActionRun，并把这些运行写入草稿 frontmatter 的 `relatedKnowledgeRunIds`、正文的“相关知识更新 ActionRun”、`## 相关知识更新` 状态/摘要表格和核对清单。该联动只把已有知识更新事实带入复盘，不自动写 Wiki/index/log、不自动确认复盘、不勾选事项尾动作。
- **计划执行知识更新后的复盘入口强化**：Workbench 计划预览会复用同一批相关知识更新事实；当同一个 `plan_execute` 已存在同源未失败/未取消的 `knowledge_rewrite` ActionRun 且仍可写复盘时，复盘按钮从“写复盘 / Write Review”切换为“复盘知识更新 / Review Knowledge Update”（`writePlanExecutionReviewWithKnowledge`），并通过提示说明复盘草稿会带入同源知识更新记录。该入口仍走 `action-run-review:<runId>`，不自动写 Wiki/index/log、不自动确认复盘、不勾选事项尾动作。
- **计划执行知识更新后的精准复盘建议**：Workbench 计划预览会通过 `getPlanExecutionKnowledgeReviewSuggestion` 读取 `findPlanExecutionKnowledgeUpdateState` 的结果；同源知识更新已写入时，复盘按钮显示“复盘知识写入 / Review Knowledge Write”，同源知识更新明确 `no_write_needed` 时，显示“复盘无需写入 / Review No-Write Decision”。该能力只让下一步复盘建议更贴近已有知识更新事实，不自动写 Wiki/index/log、不自动创建或确认复盘、不勾选事项尾动作。
- **计划执行复盘草稿显式确认**：Workbench reviews 视图现在会区分真实事项尾动作复盘和 `action-run-review:<runId>` 来源执行复盘；后者在草稿仍为 `status: draft` 时显示“确认计划执行复盘 / Confirm Plan Execution Review”（`confirmReviewSourceDraft`），调用 `confirmWorkbenchReviewDraft` 只把复盘文档改为 `status: confirmed` 并写入 `reviewedAt`，成功提示为 `reviewSourceDraftConfirmed`。该能力不会把 `action-run-review:<runId>` 冒充为事项 checklist ID，不勾选来源事项尾动作、不更新事项状态、不沉淀成果、不写 Wiki/index/log、不移动事项文件。
- **计划执行复盘确认后的后续动作文案**：Workbench 计划预览会通过 `getPlanExecutionPostReviewActionSuggestion` 读取同源复盘状态；当复盘已确认但成果仍可沉淀时，“沉淀成果 / Preserve Output”会变为“复盘后沉淀成果 / Preserve Reviewed Output”，当复盘已确认但知识仍可更新时，“更新知识 / Update Knowledge”会变为“复盘后更新知识 / Update Reviewed Knowledge”。该能力只让确认复盘后的下一步更清楚，不自动保存成果、不自动发起知识更新、不写 Wiki/index/log、不勾选事项尾动作、不更新事项状态、不移动事项文件。
- **计划执行复盘状态标签**：Workbench 计划预览通过 `findPlanExecutionReviewState` 识别同一来源事项、且 `sourceExecutionId` 或 `tailActionId` 等于 `action-run-review:<runId>` 的复盘文档；如果已有草稿则显示“复盘草稿 / Review Draft”，如果已有 `status: confirmed` 的复盘则优先显示“已复盘 / Review Confirmed”。该能力只表达既有仓库事实，不自动确认复盘、不勾选事项尾动作、不写 Wiki/index/log、不移动事项文件。
