# 3D 虚拟办公室设计

> 决定日期: 2026-06-02  
> 状态: 已确认设计，待实施计划  
> 相关页面: `src/pages/Office3DPage.tsx`  
> 设计方向: Crayfish Operations Room + 小龙虾操作员 + 任务流表演 + 渐进式 Three.js 场景

## 目标

3D 虚拟办公室不是普通状态面板，而是 OpenClaw 多 Agent 的拟人化实时空间。方案 D 确认后，办公室的主气质从“普通办公室 + 温暖小机器人”调整为 **Crayfish Operations Room**：用户从 2.5D 视角观察一个“活着的任务调度室”，小龙虾操作员在休整区、指挥工作湾、协同沙盘之间移动，用位置、动作、终端屏、状态灯和任务流节点表达 OpenClaw Gateway 中的真实运行状态。

第一版目标是做成可运行的沉浸式办公室，而不是占位页：

- 全屏 2.5D / 等距视角办公室场景。
- 工作区、会议区、休闲区三个功能区清楚可见。
- Agent 使用保留小龙虾灵魂的操作员风格。
- Agent 状态变化时有不同动线和步态。
- 悬浮信息面板只辅助解释，不抢场景主导权。

## 设计决策

### 1. 体验方向

采用沉浸式办公室优先。场景占满主内容区，用户第一眼看到的是办公室本身，而不是卡片和列表。状态面板、连接状态、Agent 详情以轻量悬浮层出现。

2.5D 是默认视角：相机固定为斜俯视，允许轻微缩放和拖拽平移，但不做复杂第一人称或自由飞行。这样既有 3D 空间感，也保持控制台的可读性。

### 2. Agent 视觉风格

Agent 是小龙虾操作员：

- 红珊瑚色甲壳、触须、分节尾部、短腿和一对可表达动作的钳子。
- 胸前或腹部有小型终端屏，状态灯使用 cyan / amber / red 表达运行、等待和异常。
- 不走真人拟人，避免恐怖谷和复杂骨骼动画。
- 每个 Agent 通过名字牌、状态灯、甲壳纹路、钳子配件和终端颜色区分。
- 动作活泼但不幼稚：钳子操作终端、拖动任务节点、围着沙盘同步、充电休整、整理工具、异常时进入检修模式。

第一版使用 Three.js 程序化几何体搭建小龙虾操作员，暂不依赖外部 GLTF 模型。后续可以替换为低多边形 GLTF 资产，但 GLTF 也必须延续小龙虾操作员语言。

### 3. 三个办公室区域

| 区域 | 对应状态 | 表现 |
|------|----------|------|
| 指挥工作湾 | 单个 Agent 有活跃会话或正在运行任务 | 小龙虾操作员在终端前用钳子操作任务卡、看日志流、偶尔检查状态灯 |
| 协同沙盘 | 多个 Agent 同时拥有活跃会话，或未来明确协作事件 | 主操作员围绕任务流沙盘拖动节点，其它 Agent 查看拓扑、点头、同步 |
| 休整区 | Agent 空闲、离线或无当前任务 | Agent 充电、喝咖啡、泡泡式 idle、整理钳子工具，多个 idle Agent 可轻微闲聊 |

Gateway 仍是事实源。Office 不发明业务状态，只把已有状态和可推断的协作模式表演出来。

### 4. 状态映射

现有 `AgentInfo.status` 只有 `idle | running | error`，且部分 Gateway 版本可能省略 `status` 字段。因此 Office 状态推导优先消费 `sessions.list` 中的真实会话状态，再使用 `AgentInfo.status` 作为回退：

| Gateway / 会话状态 | Office 状态 | 区域 |
|--------------|-------------|------|
| 单个 Agent 有 `active` 会话，或该 Agent 为 `running` | working | 工作区 |
| 多个 Agent 同时有 `active` 会话，或多个 Agent 为 `running` | collaborating | 会议区 |
| `idle` 会话、`idle` Agent，或缺省 `status` 的已存在 Agent | resting | 休闲区 |
| `error` | stuck | 工作区边缘或维修点 |
| 未连接且无法取得 Agent 数据 | offline / empty fallback | 休闲区或降级提示 |

如果会话缺少 `agentId`，Office 使用默认 Agent；没有默认 Agent 时使用列表中的第一个 Agent。会议区主讲者优先选择默认 Agent，其次选择第一个活跃 Agent。Agent 详情中的当前任务来自活跃/空闲会话标题、标签或 key。

后续如果 Gateway 提供更细粒度事件，例如任务类型、协作关系、主从 Agent、会议阶段，Office 状态机应优先消费这些显式事件。

### 5. 状态转换动画

不同状态转换要有不同气质：

- 休整区 → 指挥工作湾 / 协同沙盘：小步快跑，钳子前伸，终端状态灯亮起，有短尾迹。
- 指挥工作湾 → 协同沙盘：快步但不慌张，像把任务卡带去同步。
- 指挥工作湾 / 协同沙盘 → 休整区：动作放松，进入休整区后充电、喝咖啡或整理钳子工具。
- 错误状态：操作员停顿，终端屏闪红，钳子短暂卡住，附近出现诊断灯。

动画不直接驱动业务状态。状态变化先进入 Office 状态机，再生成从当前区域到目标区域的路径和动作。

## 技术方案

### 1. 渲染栈

采用渐进式 Three.js：

- `three` 作为核心 3D 引擎。
- React 页面中挂载一个全屏 canvas。
- 第一版直接用 Three.js API 创建场景，避免一次性引入复杂生态。
- 后续如交互复杂，可再评估 `@react-three/fiber` 和 `@react-three/drei`。

### 2. 页面结构

`Office3DPage` 拆成更小的职责单元：

| 单元 | 职责 |
|------|------|
| `Office3DPage` | 读取 store 数据，承载页面和悬浮 UI |
| `OfficeScene` | 管理 Three.js renderer、camera、scene、animation loop |
| `office-state` | 把 `AgentInfo[]` 映射为 Office Agent 状态 |
| `office-layout` | 定义三区坐标、工位、会议位、休闲位和路径点 |
| `office-actors` | 创建小龙虾操作员、终端、协同沙盘、休整道具等程序化模型 |
| `office-animation` | 插值移动、步态速度、idle 小动作、会议动作 |

每个单元要有明确接口，避免把 Three.js 初始化、Agent 映射和 UI 面板塞在一个大文件里。

### 3. 数据流

```text
Zustand useStore
  -> agents / sessions / connectionStatus
  -> deriveOfficeAgents()
  -> OfficeScene.updateAgents(nextAgents)
  -> Office state machine calculates target zones and behaviors
  -> animation loop renders positions, gestures and labels
```

React / Zustand 仍是数据源。Three.js 场景是可视化消费者，不拥有业务事实。

### 4. 场景布局

初始布局：

- 前方或入口侧：调度前台，包含接待操作员、调度柜台、抽象钳形品牌墙和入口装饰。
- 左后侧：休整区，包含沙发、扶手椅、茶几、饮水机、咖啡机、充电垫、小睡点、绿植和工具架。
- 右后侧：指挥工作湾，包含多个终端工位、日志屏幕、状态灯。
- 中后侧：协同沙盘，包含任务流节点、连接线、主操作位、听众位和诊断屏。
- 中央留出通道，所有状态转换走通道而不是瞬移。

工作区和会议区坐标使用固定布局表，按 Agent 数量分配槽位。休闲状态使用全办公室自由活动边界，不分配固定槽位。布局策略应优先把休闲 Agent 放到休息区附近，但不能把它们锁死在休息区；Agent 可以去中央通道、窗边、前台附近或空闲工位旁“摸鱼”，只要不穿过关键家具、墙体或其它 Agent。

休闲区不是状态占位符，也不能像工位一样被切成固定槽位。它是休闲状态的高倾向活动区域，而不是唯一合法区域：沙发与扶手椅表达 idle Agent 的社交和放松状态，咖啡机、饮水机、茶几、充电垫和小睡点分别承载喝咖啡、补水、短聊、充电、休眠等动作。Agent 处于休闲状态时应通过自由落点、轻量碰撞体积和避让算法决定停留位置，避免与家具或其他 Agent 重叠；如果发生碰撞，双方应出现短暂让行、后退、晃动或重新选择目标点等可见反应。后续自然语言调整办公室布局时，这些元素需要以可命名、可移动、可增删的布局对象沉淀到本地存储，而不是散落在 Three.js 渲染代码里。

3D Office 必须有地面兜底和关键碰撞体积。Agent 的世界坐标不能低于统一地面高度，动画插值、休闲动作、碰撞反应和相机刷新都不能让模型穿到地板下。墙体、前台、会议桌、工位、休息区沙发/茶几/饮水机/咖啡机/充电垫/落地灯等关键对象都应进入碰撞体集合；碰撞体可以先用轻量圆形体积近似，后续自然语言布局编辑再升级为可配置的对象包围盒。

比例基准：

- 办公室地板与三个区域优先承载 1-12 个 Agent，默认画面中区域之间要留出通道。
- 地板必须完整承载三区、前台、墙体、窗户和装饰物，不能让区域或家具像漂在地板外。
- 办公室采用“无顶房间”模型：有地板、后墙和两侧墙，保留顶部开放，方便 2.5D 相机观察。
- 墙上可以有窗户、搁板、状态灯牌等空间线索，让用户理解这是办公室而不是几块浮动平台。
- 前台后方必须有一面承载品牌信息的墙，墙面展示当前实例的公司名或团队名。
- 程序化小龙虾操作员应明显小于工位与协同沙盘，避免“模型过大导致办公室像玩具盒”。
- 调整比例时同时考虑布局坐标、相机正交视锥和小龙虾操作员基础缩放，不能只改一个参数。

### 5. 主题与画面风格

3D Office 必须跟随应用主题，而不是固定暗色。`Office3DPage` 从设置读取有效主题模式和主色，生成 `OfficeTheme` 后传给 Three.js 场景。

| 应用主题 | Office 表现 |
|----------|-------------|
| 浅色 | 浅地面、浅雾色、半透明白色面板，保留蓝/绿/橙三区识别 |
| 暗色 | 深色地面、暗雾色、半透明深色面板，保持现有沉浸感 |
| 主色 | 用于屏幕高亮、图标和部分区域强调，跟随设置中的主题色 |

Three.js 代码不应散落硬编码全局色值；场景背景、地板、家具、标签、面板文字都应来自 Office 主题调色板。

场景颜色不能只做“浅色/暗色同色系替换”。工作区、会议区、休闲区必须有足够明显的色相差异，地板、区域、家具、黑板/屏幕之间也要有亮度差。每个区域应有透明色块和边线双重提示，地板可使用低透明网格帮助用户判断空间深度和边界。

雾效要谨慎：浅色主题默认关闭相机雾，避免远景灰蒙蒙；暗色主题只在很远处使用轻微雾效，制造深度但不能影响读图。

### 6. 交互

第一版交互保持轻量：

- 点击 Agent：显示悬浮详情，包括名称、状态、模型、当前任务摘要。
- 点击区域：突出该区域的 Agent 列表。
- 点击前台：接待 NPC 介绍当前连接状态、Agent 数、会话数和三区人数。
- 刷新按钮：调用 `refreshAll()`。
- 连接断开：办公室保留但降低亮度，Agent 进入 offline/resting 表现。
- WASD：按当前相机朝向在水平面平移视角。
- 鼠标中键拖拽：围绕当前观察目标旋转相机。
- 鼠标滚轮：改变相机距离和正交视锥，实现靠近/远离。
- 一键还原：恢复默认 2.5D 视角、距离和观察目标。
- 空格键：同样恢复默认 2.5D 视角。
- 鼠标左键拖动：按当前相机朝向平移观察目标，行为与 WASD 平移一致。
- 选中 Agent 后：WASD 不再移动整体相机，而是锁定并驱动当前选中的小龙虾操作员在办公室内模拟行走；相机切到该 Agent 头部附近的第一视角，并与该 Agent 本体同步移动，行走速度应明显快于普通休闲游走，行走时需要有钳子摆动和身体轻微晃动。
- Q 键在第一视角切换本地诊断脉冲工具；左键发射诊断脉冲，命中其它本地角色时显示本地状态反馈、状态缓冲条、临时 down/recover 循环。这个玩法只影响本地表现，不写回 Gateway，不改变真实 Agent 状态。
- V 键：退出选中 Agent 的第一视角操控，清空操控对象并恢复默认第三人称 2.5D 视角。

暂不做拖拽 Agent、自由编辑布局、复杂相机路线。

### 7. 本地办公室资料与自然语言布局基础

3D Office 的展示资料属于实例级本地扩展数据，使用既有文件存储逻辑按实例保存：

| key | 职责 |
|-----|------|
| `office-profile` | 公司名、前台问候语等展示资料 |
| `office-layout-instructions` | 未来自然语言调整布局的指令文本、状态、摘要和应用时间 |

当前版本先消费 `office-profile.companyName` 渲染前台品牌墙；如果实例没有该文件，则从当前实例名称生成默认值并写入实例目录。后续做自然语言布局调整时，用户表达如“把会议区挪到靠窗的位置”应先转成结构化布局指令，保存为 `OfficeLayoutInstruction`，再由布局解释器更新三区坐标、家具和装饰配置。

## 错误处理

- WebGL 不可用时，显示降级版 2D 状态列表和错误提示。
- Gateway 未连接时，显示离线办公室，不空白。
- Agent 数据缺字段时使用 `agent.id` 和默认小龙虾操作员外观。
- Three.js 初始化或 animation loop 错误不能阻断整个应用，应清理 renderer 并显示降级内容。

## 性能约束

- 第一版目标 Agent 数量：1-12 个。
- 使用程序化低复杂度几何体，避免大贴图和外部模型。
- animation loop 在页面不可见或组件卸载时停止。
- renderer resize 使用节流或 ResizeObserver。
- 文本标签用 HTML overlay 或轻量 sprite，避免大量动态纹理。

## 测试与验证

实现时需要覆盖：

- `office-state` 单元测试：Agent 状态到区域/行为映射。
- `office-layout` 单元测试：不同 Agent 数量分配稳定槽位。
- 页面 smoke test：未连接、有 idle、有 running、有 error 时页面不崩。
- 视觉验证：用浏览器 / Playwright 打开本地页面，确认 canvas 非空、三区可见、Agent 位置正确、窗口 resize 后不变形。
- 构建验证：`npm run typecheck`、相关测试、`npm run build`。

## 非目标

第一版不做：

- 外部 GLTF 角色资产和骨骼动画。
- 自由相机漫游或第一人称办公室。
- 用户自定义摆放家具。
- 多人在线共享同一办公室视角。
- 将 Office 状态写回 Gateway。

## 后续演进

1. 增加显式协作事件来源，让会议区不只依赖启发式推断。
2. 替换程序化小龙虾操作员为低多边形 GLTF 模型。
3. 增加黑板内容：当前任务、计划、工具调用摘要。
4. 增加 Agent 个性化：颜色、配件、表情、常驻工位。
5. 增加录制/回放：展示一段 Agent 协作过程。
